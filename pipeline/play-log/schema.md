# Schema — The Play Log

## Path
Greenfield

## Overview
Two tables on Neon Postgres (via Drizzle ORM): `plays` holds one row per
logged play, and `play_actors` holds the ordered, per-entry-deduplicated
actor names for each play. This supports fast entry, list/sort/search,
exact-match click-to-filter on venue, director, and actor, and clean
CSV import/export later.

## Tables / Models

### plays
One row per logged play entry. All optional free-text fields are stored
as `NULL` when empty (never empty string), so "blank" is unambiguous in
display and sorts blanks-last cleanly.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| id | uuid | PRIMARY KEY, DEFAULT gen_random_uuid() | Auto-generated |
| name | text | NOT NULL, CHECK (btrim(name) <> '') | Play name (FR-02). Trimmed before write (FR-06); check is a backstop |
| date_seen | date | NULL | Calendar date only, no time zone semantics (FR-04). Past or future accepted |
| venue | text | NULL | Trimmed; NULL when empty (FR-03) |
| director | text | NULL | Trimmed; NULL when empty (FR-03) |
| created_at | timestamptz | NOT NULL, DEFAULT now() | System-generated |
| updated_at | timestamptz | NOT NULL, DEFAULT now() | Set to now() on every update (app-level via Drizzle) |

**Relationships:**
- `plays` has many `play_actors` via `play_actors.play_id` (one-to-many, ON DELETE CASCADE).

**Indexes:**
- `(date_seen DESC NULLS LAST)` — default list order is date seen, newest first, blanks last (FR-11). `NULLS LAST` must be explicit: Postgres defaults DESC indexes/sorts to nulls-first.
- `(venue)` — exact-match click-to-filter on venue (FR-15).
- `(director)` — exact-match click-to-filter on director (FR-15).

### play_actors
Ordered actor names for a play. Modeled as a child table (not a text[]
column) so actor filtering is a plain indexed equality join, per-entry
duplicate prevention is a database constraint, and CSV export later is a
straightforward join.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| id | uuid | PRIMARY KEY, DEFAULT gen_random_uuid() | Auto-generated |
| play_id | uuid | NOT NULL, REFERENCES plays(id) ON DELETE CASCADE | Deleting a play removes its actors (FR-09) |
| name | text | NOT NULL, CHECK (btrim(name) <> '') | Exact stored text; trimmed before write, blanks discarded (FR-06) |
| position | integer | NOT NULL | 0-based entry order (FR-05: order of entry preserved) |

**Constraints:**
- UNIQUE `(play_id, name)` — at most one copy of an exact-duplicate actor name per entry (FR-06).
- UNIQUE `(play_id, position)` — stable, unambiguous ordering per entry.

**Relationships:**
- `play_actors` belongs to `plays` via `play_id`.

**Indexes:**
- `(name)` — exact-match click-to-filter on actor (FR-15): entries where *any* actor matches resolve via an indexed equality lookup then join to `plays`.
- `(play_id)` — covered by the UNIQUE `(play_id, name)` index; supports fetching a play's actors and cascade deletes. No separate index needed.

## Query Patterns the Schema Serves

- **Default list (FR-10/FR-11):** select `plays` LEFT JOIN `play_actors` (or a second query batched by play_id), `ORDER BY date_seen DESC NULLS LAST`. Actors ordered by `position`.
- **Sort by name / venue (FR-11):** `ORDER BY name ASC|DESC` / `ORDER BY venue ASC|DESC NULLS LAST` (explicit `NULLS LAST` in both directions).
- **Click-to-filter (FR-15/FR-17):** venue/director → `WHERE venue = $1` / `WHERE director = $1` (exact, case-sensitive, matches stored text). Actor → `WHERE EXISTS (SELECT 1 FROM play_actors pa WHERE pa.play_id = plays.id AND pa.name = $1)`.
- **Free-text search (FR-13):** case-insensitive substring via `ILIKE '%q%'` across `name`, `venue`, `director`, and an EXISTS over `play_actors.name` — combined with an active filter via AND (FR-17). At the NFR-03 scale (≤1,000 entries) sequential ILIKE is comfortably imperceptible; no trigram index in v1.
- **Edit (FR-08):** update `plays` row; replace the entry's `play_actors` set (delete + reinsert with fresh positions) inside one transaction — simpler and safer than diffing, trivial at this row count.

## Migration Plan

Single initial migration (`0001_init`, generated by `drizzle-kit`), in order:
1. Create `plays` with columns, checks, and defaults (`gen_random_uuid()` — pgcrypto is built-in on Neon PG15+; no extension statement needed).
2. Create `play_actors` with FK (ON DELETE CASCADE), checks, and both UNIQUE constraints.
3. Create indexes: `plays(date_seen DESC NULLS LAST)`, `plays(venue)`, `plays(director)`, `play_actors(name)`.

No seed data. No destructive steps — greenfield, nothing exists yet.
Rollback = drop both tables.

## Design Decisions

- **Child table over `text[]` for actors** — indexed exact-match actor filtering, DB-enforced per-entry dedupe, preserved order via `position`, and a clean relational shape for the CSV import/export roadmap item. An array column would push dedupe and filtering into app code for no gain.
- **No canonical people/venue entities** — the brief and PRD fix v1 as free text with exact-match filtering; a `people` table would add joins and merge semantics the product explicitly defers. Filter values are just stored strings.
- **NULL, not empty string, for optional fields** — makes FR-03 (blank display, no fake placeholders) and FR-11 (blanks last in both sort directions) mechanical rather than special-cased.
- **`date` type, not `timestamptz`, for date_seen** — the PRD specifies a calendar date; storing a timestamp would invite timezone-shift bugs ("saw it on the 14th" showing as the 13th).
- **Filtering is case-sensitive exact match** — per the PRD's out-of-scope list; the `(venue)`, `(director)`, `(name)` btree indexes serve it directly.
- **Normalization (trim, discard-empty, dedupe) happens at the app layer on save** (FR-06), with CHECK and UNIQUE constraints as database backstops so bad data can't slip in through any path.
- **IMDb links need no schema support** — they are URL-encoded stored names rendered at display time (FR-18); nothing extra is persisted.
- **Auth: none in v1** — single-user personal app; no user table, no `user_id` columns. Access control is deployment-level (keep the deployment private / enable Vercel Deployment Protection). If accounts ever arrive, adding a `user_id` to `plays` is a small additive migration.

## Assumptions

- One deployment = one user; concurrent multi-user writes are not a design constraint (last-write-wins on edit is acceptable).
- Actor lists are small (typically < 30 per play); delete-and-reinsert on edit is fine.
- "Exact match" means byte-for-byte equality on stored (trimmed) text, including case — "ian mckellen" and "Ian McKellen" are different values by design.
- The 1,000-entry NFR-03 bound is the realistic ceiling; no pagination, materialized search, or trigram indexing is warranted in v1.
- Future CSV import/export will map one row per play with actors joined/split by a delimiter — this schema round-trips that without change.
